The SPARK Group was founded in 2024 as an informal open-source collective, bringing together a small group of people before I departed for university. The name is an acronym: Strategic Pursuits in the Advancement of Research and Knowledge. In practice, the group functions as a personal research and development lab — a space to build, break, and learn from the ground up.

I am not a technical person by nature. Every project here represents a deliberate act of learning: choosing hard problems, building from first principles, and documenting the process honestly. The SPARK Group is less a finished product and more a philosophy of inquiry applied to software, hardware, and networks.

All projects are open source and hosted on the SPARK Group GitHub. Development is ongoing and active, though the pace is shaped by academic commitments.

The SPARK Group operates under its own bespoke licensing agreement, drafted as part of Project Synthacope. All outputs are open source within the terms of that agreement.

Project II — SenaCode & SCOS
A Sovereign Operating Environment
Ongoing Systems OS

SenaCode is a minimalist operating environment designed for the Raspberry Pi 5, built around a single guiding principle: Security through Simplicity. Modern operating systems contain tens of millions of lines of code, written by thousands of contributors — so complex that no single person can fully understand, let alone audit, them. SenaCode is a direct challenge to that assumption.

The project has three interlocking components. SCOS (SenaCode OS) is the bare-metal operating system, written in C with elements of ARM assembly for boot loading, running entirely without unnecessary dependencies. GATBUI (Graphically Assisted Text-Based User Interface) is the interface layer — a full-screen terminal environment inspired by retro computing aesthetics (think Pip-Boy from Fallout, or DOS). It is divided into four ASCII-bounded decks: an upper info panel, a main display area, a persistent console input, and a live system stats bar. SenaLang is a bespoke general-purpose programming language, modelled after BASIC — readable, auditable, and sandboxed. Every line of SenaLang code can be understood and verified by a single person.

The security model is deliberately architectural rather than procedural. System-level files are logically separated from user-accessible ones — internally referred to as the "drinks cabinet" (read-only) and "snacks cupboard" (user space). An integrated audit log records all commands. The goal is not to compete with Linux; it is to prove that a complete, functional, and secure computing environment can be built lean.

Stack

C; ARM Assembly; custom SenaLang interpreter; Raspberry Pi 5

Key Concepts

Bare-metal OS design; security through simplicity; auditable computing; bespoke language design

Project III — SparkNet
A Decentralised Sovereign Network
Ongoing Networks P2P

SparkNet is the project I have invested the most research into. It began as a practical need — secure communication between geographically dispersed SPARK Group members — and evolved into a full theoretical and practical framework for a sovereign, decentralised network layer that operates entirely outside the conventional internet infrastructure.

At the core of SparkNet is the SNX (SenEx) Protocol — a self-contained, identity-first routing and trust system that eliminates reliance on IP addresses, DNS, and centralised authorities like ICANN. Instead, nodes are identified via SNX-IDs in the format NodeType:OrgName:Code/Path (e.g. MAC:Manus:101/home). These identifiers encode not just destination but organisational affiliation and node purpose, enabling routing and access control without any third-party lookup service.

Trust within SparkNet is managed through a ticket-based model: cryptographically signed Certs and Tickets govern every meaningful interaction — from resource access to data transactions. The Root Command Server (RCS) acts as the certificate authority for the network, while Centrally Administered Systems (CASs) function as organisational root nodes — each effectively a digital nation-state, governing its own nodes and infrastructure.

For external communication, SparkNet deploys EXT (Gateway) nodes that bridge the internal network to the MainNet via TOR tunnels — preserving anonymity and ensuring SNX nodes are never exposed to the ClearNet. Crucially, SparkNet is also designed for entirely off-grid operation: nodes can communicate via USB, Ethernet, or serial links, and because SNX packets are self-contained and verifiable, they can be physically transported on USB drives and re-injected into the network. This enables delay-tolerant networking in environments with no internet access at all.

Key Concepts

Decentralised networking; sovereign protocols; TOR integration; ticket-based authentication; delay-tolerant networking

Network Components

RCS (Root Command Server); CAS (Centrally Administered System); EXT (Gateway); DAN (Direct Allied Network); USR (User Node)

Project IV — Kielder
Legacy Systems & Intelligence Reservoirs
Complete Database COBOL

Named after Kielder Water — a reservoir — Project Kielder explores legacy systems and intelligence data management. Conducted as part of my Integrated Industry Experience module at the University of Buckingham's School of Computing, the project produced a COBOL-based system capable of indexing and managing report files from external hardware (primarily USB drives).

The system integrates COBOL with modern ZSH shell scripting to automate data ingestion — reading structured headers from files (including WordGrinder documents, PDFs, and plain text), extracting metadata, and maintaining a flat-file index. The project exists in three versions: TryNull (the first working front-end, pure COBOL), TryOne (extended features, still pure COBOL), and the unreleased TryTwo, which integrates embedded SQL for genuine database functionality and folder access.

The project also includes a technically earnest self-destruct function — a data purging routine included as an exercise in secure deletion. The work on Kielder directly led to broader engagement with IBM Mainframe systems and globally recognised certifications in that area.

Stack

COBOL; ZSH; embedded SQL (TryTwo); Raspberry Pi hardware integration

Outputs

Working COBOL indexing system; USB data ingestion pipeline; IBM Mainframe certifications

Project V — Clifford
Hardware Integration & Physical-Digital Interface
Theoretical Hardware

The final SPARK Group project, and the one I have been least directly involved in. Clifford is named as a pun on Clifford the Big Red Dog — the "Big Red Button" being the central conceit of the project. It explores the physical-digital interface: the point at which software meets hardware in a tangible, user-facing way.

Conceptually, Clifford is the hardware counterpart to SenaCode's software. Where SenaCode defines the operating environment, Clifford defines the physical device that runs it — exploring how bespoke hardware design can reinforce and complement a security-first software philosophy. My involvement has been primarily in the theoretical framing of the physical-digital interface rather than the hands-on hardware development.

Focus

Physical-digital interface; hardware design philosophy; SenaCode integration

Status

Theoretical exploration; active development by other group members
